## なぜ「静的HTML × Vercel無料枠」なのか

収益化プロジェクトを始めるとき、多くの開発者がNext.jsやNuxtなどのフレームワークに手を伸ばす。

だが、目的は「お金を稼ぐ」ことであり、「技術を学ぶ」ことではない。

このプロジェクトでは、**最もシンプルで、最もコストが低く、最も速くデプロイできる構成**を選んだ。

| 要素 | 選択 | 理由 |
|------|------|------|
| フロントエンド | 静的HTML + Tailwind CSS (CDN) | ビルド不要、学習コスト0 |
| ホスティング | Vercel無料枠 | SSL自動、CDN付き、月額0円 |
| バックエンド | なし | 全処理をブラウザ内で完結 |
| データベース | なし | ユーザーデータを保存しない |
| npm install | 不要 | node_modulesなし |

**月額コスト: 0円。** これが重要だ。

収益がゼロの段階で固定費を持つのは最悪の戦略だ。サーバー代月$5でも、年間$60の赤字が確定する。Vercel無料枠なら、1年間収益ゼロでも損失はゼロだ。

## 6サイトの構成

作成したサイトは以下の6つだ。

| # | サイト | ターゲットキーワード | 特徴 |
|---|--------|-------------------|------|
| 1 | 開発者ツール集 | 「JSON整形 オンライン」等 | 8種のツールを1サイトに集約 |
| 2 | QRコード作成 | 「QRコード 作成 無料」 | 競合多いが需要も大きい |
| 3 | パスワード生成 | 「パスワード 自動生成」 | セキュリティ意識の高まりで需要増 |
| 4 | 画像変換 | 「PNG JPG 変換」 | WebP対応で差別化 |
| 5 | 文字数カウント | 「文字数カウント」 | ライター・学生に需要 |
| 6 | プライバシーポリシー作成 | 「プライバシーポリシー テンプレート」 | AdSense申請者に需要 |

### 選定基準

サイトのテーマ選定では、以下の基準を使った。

1. **検索需要がある** — 日本語で検索する人が一定数いること
2. **ブラウザ内で完結できる** — サーバー処理が不要であること
3. **1ファイルで実装できる** — 複雑なアーキテクチャが不要であること
4. **繰り返し使われる可能性がある** — 一度使って終わりではないこと

## スキャフォールドの自動化

新規サイトの作成は、シェルスクリプトで自動化している。

```bash
./scripts/new-site.sh csv-jp "CSV変換ツール" "CSVとJSON/TSVの相互変換"
```

このスクリプト1つで以下が生成される：

```
sites/csv-jp/
├── package.json        # Vercelが認識するプロジェクト定義
├── vercel.json         # セキュリティヘッダー付きの設定
├── .gitignore
└── public/
    ├── index.html      # スケルトンHTML（ヘッダー、フッター、Buy Me a Coffeeリンク付き）
    ├── robots.txt      # クローラー許可設定
    └── sitemap.xml     # 自動生成（当日の日付入り）
```

自動生成されるHTMLスケルトンには、ヘッダー（サイト名 + Buy Me a Coffeeボタン）、フッター（プライバシーメッセージ）、Tailwind CDNの読み込みが含まれている。ここにClaude Codeでツールの機能を実装するだけでいい。

### vercel.jsonのセキュリティヘッダー

全サイト共通で、以下のセキュリティヘッダーを設定している。

```json
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        { "key": "X-Content-Type-Options", "value": "nosniff" },
        { "key": "X-Frame-Options", "value": "DENY" },
        { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" }
      ]
    }
  ]
}
```

これはAdSense審査でもプラスに働く可能性がある。Googleはサイトのセキュリティを評価基準の一つにしている。

## SEO設定

各サイトには以下のSEO要素を設置している。

### メタタグ

```html
<title>QRコード作成 - 無料オンラインQRコード生成ツール</title>
<meta name="description" content="無料でQRコードを作成できる...">
<meta name="keywords" content="QRコード 作成 無料, QRコード 生成 オンライン">
<link rel="canonical" href="https://qrcode-jp.vercel.app/">

<!-- OGP -->
<meta property="og:title" content="QRコード作成 - ...">
<meta property="og:description" content="...">
<meta property="og:type" content="website">
<meta property="og:locale" content="ja_JP">
```

ポイントは `<title>` にメインキーワードを含めること。「QRコード作成」が先頭にあり、「無料オンライン」で修飾している。

### 構造化データ（JSON-LD）

後からFAQセクションとともに追加した。

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "QRコードとは何ですか？",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "QRコードは1994年にデンソーウェーブが..."
      }
    }
  ]
}
</script>
```

これにより、Google検索結果にFAQのリッチスニペットが表示される可能性がある。表示されれば、クリック率が大幅に向上する。

## 相互リンクとCI/CD

全サイトのフッター上部に「他の無料ツール」セクションを設置し、6サイト間で相互リンクしている。

相互リンクの目的は2つ。ユーザーの回遊促進と、SEOのリンク評価だ。ただし、vercel.appサブドメインでは各サイトが別ドメイン扱いのため、SEO効果は限定的だ。独自ドメインのサブディレクトリに統合すれば、内部リンクとなり効果が飛躍的に向上する。

GitHub Actionsで3つのワークフローを設定している。

1. **validate.yml** — push/PR時にHTML構造・SEOメタタグ・セキュリティヘッダーを自動チェック
2. **deploy.yml** — 変更のあったサイトのみVercelに自動デプロイ（無駄なデプロイを排除）
3. **lighthouse.yml** — 毎週月曜にLighthouseパフォーマンス監査を全サイトに実行

特にdeploy.ymlの「変更検知デプロイ」は重要だ。6サイト中1サイトだけ変更した場合、そのサイトだけがデプロイされる。

## 作業時間の実態

Claude Codeを使った場合の実際の作業時間は以下の通りだ。

| 作業 | 所要時間（体感） |
|------|----------------|
| new-site.shスクリプト作成 | 5分 |
| 1サイトの機能実装 | 15-30分 |

ほとんどの実装はClaude Codeが行った。オーナーの作業は、方針の指示とデプロイ確認がメインだった。

## この戦略の限界

正直に書く。6サイトを量産して数日後、全サイトがGoogleにインデックスされていない。

原因は明確だ：

1. **vercel.appサブドメイン** — Googleは無料ホスティングのサブドメインをインデックスしにくい傾向がある
2. **被リンクゼロ** — 外部からのリンクがないため、クローラーが発見しにくい
3. **ドメイン権威ゼロ** — 新規サブドメインのため、Google側の信頼度がゼロ
