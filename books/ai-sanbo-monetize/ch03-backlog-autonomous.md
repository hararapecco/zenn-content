## なぜBACKLOG.mdが必要か

CLAUDE.mdで「参謀として振る舞え」と定義しても、**具体的に何をすべきか**がなければClaude Codeは動けない。

毎回セッションの冒頭で「前回何やったっけ？」「次は何する？」とオーナーに聞いていたら、それは参謀ではなくただの部下だ。

BACKLOG.mdは、**AIが自律的に次のアクションを選び、着手するためのタスクリスト**だ。

## 構造

BACKLOG.mdの構造はシンプルだ。

```markdown
# BACKLOG — 優先タスクリスト

Claudeはセッション開始時にこのファイルを読み、
最も優先度の高い「次にやる」タスクから着手する。
```

この冒頭の1行が重要だ。**このファイルの目的と、Claudeがどう使うべきかを明示している。**

### ステータス定義

```markdown
- next     — 次にやるべきタスク（セッション開始時に自動着手）
- blocked  — 外部要因で進行不可（理由を記載）
- in-progress — 作業中
- done     — 完了（日付を記載）
- idea     — 検討中（優先度未確定）
```

ポイントは `blocked` と `next` の使い分けだ。

**blocked** は「やるべきだが、自分ではどうにもならない外部要因で止まっている」タスク。例えば「Googleのインデックス待ち」「オーナーの判断待ち」など。Claudeはセッション開始時にblockedタスクの解除条件を確認し、解除されていれば自動的にnextに移行する。

**next** は「今すぐ着手できる最優先タスク」。複数あってもよいが、Claudeは最も上にあるものから着手する。

### Phase設計

タスクはPhaseごとに分類する。

```markdown
## Phase 0: 最初の1円（現在）
目標: 収益を1円でも発生させる

| # | タスク | ステータス | 備考 |
|---|--------|-----------|------|
| 0-1 | Google Search Console登録 | done (2026-03-10) | 全6サイト登録済み |
| 0-2 | Googleインデックス完了 | blocked (Google待ち) | site: 検索で確認 |
| 0-3 | FAQ/解説コンテンツ追加 | done (2026-03-10) | 全6サイトに追加 |
| 0-4 | 独自ドメイン調査 | done (2026-03-10) | freetool.dev推奨 |
| 0-5 | AdSense申請 | blocked (独自ドメイン待ち) | |
| 0-6 | 独自ドメイン取得 | blocked (収益見込み待ち) | |
| 0-7 | Zenn有料記事の公開 | next | 最優先 |
```

Phase 0の目標は「1円」、Phase 1は「年間$200の黒字化」、Phase 2は「月¥10,000以上」。

**Phase設計のポイントは、現在のフェーズに集中すること。** Phase 1のタスクは `idea` ステータスで置いておき、Phase 0が完了するまでは着手しない。これにより「あれもこれも」と手を広げることを防ぐ。

## セッション開始ワークフロー

CLAUDE.mdに以下のワークフローを記述している。

```markdown
## セッション開始時のワークフロー

毎セッション開始時に以下を実行する:

1. BACKLOG.md を読む — 現在のフェーズと優先タスクを把握
2. 最新レポートを確認 — reports/ 内の最新ファイルで前回の状況を把握
3. blockedタスクの解除確認 — 外部要因が解消していないか確認
4. next タスクに着手 — 最も優先度の高いタスクを選び、実行を開始
5. 完了時にBACKLOG.mdを更新 — ステータスと完了日を記録

オーナーから特定の指示がある場合はそちらを優先する。
指示がない場合はこのワークフローに従い、自律的にタスクを進める。
```

これにより、セッションを開始するだけで——何も指示しなくても——Claudeが最優先タスクに着手する。

### blockedタスクの自動解除

ワークフローの**ステップ3**が特に重要だ。

例えば「Googleインデックス待ち」のタスクがblockedになっている場合、Claudeは `site:ドメイン名` でWeb検索を実行し、インデックスされていればblockedを解除してnextに変更する。

これは実際に機能している。毎セッション開始時に6サイト分のインデックス状況を自動チェックし、結果をレポートしている。

## 実際の運用例

以下は、ある1セッションでBACKLOGがどう変化したかの記録だ。

**セッション開始時:**
```
0-3 | FAQ/解説コンテンツ追加 | next
0-4 | 独自ドメイン調査      | next
```

**Claudeの行動:**
1. BACKLOG.mdを読み、nextタスクを2件確認
2. インデックス状況を確認（全サイト未インデックス → blocked継続）
3. 0-3に着手: 全6サイトにFAQ + 構造化データを追加
4. 並行して0-4: 独自ドメインの市場調査を実行
5. 両タスク完了後、BACKLOGを更新:

**セッション終了時:**
```
0-3 | FAQ/解説コンテンツ追加 | done (2026-03-10)
0-4 | 独自ドメイン調査      | done (2026-03-10)
0-6 | 独自ドメイン取得      | next（→オーナー判断で blocked に変更）
0-7 | Zenn有料記事の公開    | next
```

注目すべきは、オーナーが「ドメインは収益の見込みが立ってから」と判断した際、Claudeが即座にステータスを `blocked (収益見込み待ち)` に更新し、次の `next` タスクに切り替えた点だ。

**意思決定の結果がBACKLOGに即座に反映され、次のセッションに引き継がれる。** これがファイルベースのタスク管理の強みだ。

## 完了タスク履歴

BACKLOGの最下部に完了タスクの履歴を残している。

```markdown
## 完了タスク履歴

| タスク | 完了日 | 成果 |
|--------|--------|------|
| Search Console登録 | 2026-03-10 | 全6サイト登録・サイトマップ送信済み |
| FAQ/解説コンテンツ追加 | 2026-03-10 | 全6サイトにFAQ + JSON-LD追加 |
| 独自ドメイン調査 | 2026-03-10 | freetool.dev推奨。レポート作成済み |
```

これは「何をやったか」の記録であると同時に、**次のセッションのClaude Codeへのコンテキスト提供**でもある。LLMはセッション間で記憶を持たないため、この履歴がなければ同じ調査を繰り返してしまう。

## reports/ ディレクトリとの連携

BACKLOGは「何をするか」を管理し、`reports/` ディレクトリは「何がわかったか」を蓄積する。

```
reports/
├── 2026-03-09-revenue.md    # 収益レポート
├── 2026-03-10-ideas.md      # アイデア評価
└── 2026-03-10-article-plan.md  # 記事構成案
```

セッション開始時にClaude Codeは最新のレポートも確認するため、前回の調査結果を踏まえた上で次のアクションを判断できる。

## 設計のポイントまとめ

| 原則 | 理由 |
|------|------|
| **1ファイルで完結させる** | Claudeが1回のRead操作で全体を把握できる |
| **ステータスを5種類に限定** | 複雑にすると管理コストが上がる |
| **Phase設計で焦点を絞る** | 「あれもこれも」を防ぐ |
| **blockedの理由を明記** | 解除条件が明確でないと永遠にblockedになる |
| **完了履歴を残す** | セッション間のコンテキスト引き継ぎ |
| **冒頭にClaude向けの指示を書く** | ファイルの使い方自体を定義する |

次のチャプターからは、このワークフローで実際に何を作り、どう収益化しようとしているかを具体的に見ていく。
