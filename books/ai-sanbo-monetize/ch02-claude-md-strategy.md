## 普通のCLAUDE.mdの問題

多くのプロジェクトで、CLAUDE.mdは**技術仕様書**になっている。

```markdown
## アーキテクチャ
- フレームワーク: Next.js 15
- スタイル: Tailwind CSS
- DB: PostgreSQL

## コーディング規約
- 変数名: camelCase
- コンポーネント: PascalCase
```

これは悪くない。Claudeがコードを書くときに一貫性を保つために必要な情報だ。

しかし、**収益化を目的とするプロジェクト**では、これだけでは致命的に不足している。

なぜなら、CLAUDE.mdが技術仕様だけを記述している場合、Claudeは「指示されたものを作る実装者」としてしか機能しない。**何を作るべきか、なぜ作るのか、いつやめるのか**を判断する基準がない。

## Before: 実装者を縛るCLAUDE.md

当プロジェクトの初期のCLAUDE.mdは、まさにこの問題を抱えていた：

```markdown
## プロジェクトの目的
**お金を稼ぐこと。** これが唯一の目的であり、判断基準である。

## 現在の手段と方針

### アーキテクチャ
- 全サイトは **静的HTML + Tailwind CSS CDN** で構成する
- 全処理はクライアントサイドで完結させる
- 外部ライブラリはCDN経由のみ使用可
- Vercel無料枠でホスティングする

### 新規サイト追加時の手順
1. sites/<site-name>/ ディレクトリを作成
2. public/index.html に全機能を実装
3. vercel.json を作成
...（10ステップの手順が続く）
```

一見よさそうに見える。「お金を稼ぐ」という目的も書かれている。

しかし問題は3つある：

**1. 目的が抽象的すぎる**

「お金を稼ぐ」は目的だが、いくら稼ぐのか、いつまでに稼ぐのかが不明確。Claudeは「月1万円」と「月100万円」で同じ戦略を取ってしまう。

**2. 手段が固定されている**

「静的HTML + Tailwind + Vercel」というアーキテクチャが詳細に規定されている。「手段に過ぎない」と書きつつ、実質的にClaude Codeの行動をツールサイト量産に縛っている。

**3. 「何を作るか選ぶ」がない**

判断の優先順位は「収益に直結するか」から始まるが、これは**すでに決まった手段の中での優先順位**であり、「そもそも何が最も稼げる手段か」を探索する指針がない。

## After: 参謀が動くCLAUDE.md

書き直した後のCLAUDE.mdは、構造が根本的に変わった。以下が主要な変更点だ。

### 1. 目的を具体的にする

```markdown
## プロジェクトの目的
**お金を稼いで黒字化すること。**

- 最初の目標: まず1円の収益を発生させる
- 当面の目標: Claude年間課金 $200（約30,000円）を超える収益を得て黒字化する
- 時間軸: できるだけ早く
```

「まず1円」→「年間$200を超える」という段階的な目標設定。Claudeはこの具体的な数値を基準に、戦略の規模感を判断できる。月100万円を狙う必要がないことも、ここから読み取れる。

### 2. Claudeの役割を明示する

```markdown
## Claudeの役割
**主体的に稼ぐ方法を考え、提案し、実行する戦略パートナー。**

- 市場調査・戦略立案・実装・検証まで主導する
- 「指示を待つ実装者」ではなく「収益を生む方法を自ら探索する参謀」として振る舞う
- 開発作業を他のAIエージェントに委託する提案も可
- オーナーの手作業は最小限に。必要な場合は具体的に依頼する
```

これが最も重要な変更だ。**役割を明示的に「参謀」と定義する**ことで、Claudeの振る舞いが変わる。

指示がなくても市場調査を始める。「これを作って」と言われなくても、「これは作るべきではない」と提案する。セッション開始時に自分でタスクを選び、着手する。

### 3. 判断の最上位を変える

```markdown
## 判断の優先順位
1. **何が最も稼げるかを選ぶ**（手段の探索・選択・切り替え）
2. **収益に直結する作業を優先する**（決済導線、集客、コンバージョン）
3. **収益の基盤を強化する**（SEO、品質、信頼性）
4. **運用効率を上げる**（自動化、CI/CD）
5. 上記いずれにも該当しなければ、やらない
```

旧版との決定的な違いは**1番目**だ。「何が最も稼げるかを選ぶ」が最上位にある。これにより、Claudeは「今の手段（ツールサイト量産）は本当に最適か？」を常に問い続ける。別の手段（Chrome拡張、有料記事、Micro-SaaS等）のほうが有望なら、切り替えを提案する。

### 4. 技術制約を撤廃する

```markdown
## 制約条件

### やってよいこと
- 日本の法律・倫理・公序良俗に反しない収益手段はすべて検討可
- 技術スタック・コンテンツ内容・マネタイズ手法に制限なし
```

旧版で詳細に規定していた「静的HTML + Tailwind + Vercel」は、「既存サイトの技術仕様（参考）」セクションに移動した。新しいプロダクトは別の技術で作ってもいい。手段は目的に従う。

### 5. 投資の判断基準を明確にする

```markdown
### 投資の判断基準
- 無料で始められる手段を優先する
- 金銭的投資はClaude年間$200を目安に、黒字化の見通しがあれば許容
- それを超える投資は要相談
```

これにより、Claudeは「ドメインを取得すべきか？」「有料APIを使うべきか？」といった投資判断を、オーナーの基準に沿って自律的に行える。

## 設計のポイント

CLAUDE.mdを戦略文書として設計する際のポイントをまとめる。

### 書くべきこと

| セクション | 内容 | なぜ必要か |
|-----------|------|-----------|
| 目的 | 具体的な数値目標と時間軸 | 戦略の規模感を決める |
| Claudeの役割 | 実装者か、参謀か、自律的パートナーか | 振る舞いの根本を定義する |
| 判断の優先順位 | 何を最上位に置くか | すべての意思決定の基準 |
| 制約条件 | やってよいこと、投資の上限 | 自律的に動ける範囲を定義する |
| 現在の資産 | 既存プロダクト、アカウント、インフラ | 活用できるリソースの棚卸し |

### 書くべきでないこと

- **手段を固定する詳細な技術仕様**（参考として別セクションに置く）
- **「〜してはいけない」の羅列**（制約が多すぎると自律性を殺す）
- **曖昧な目的**（「お金を稼ぐ」だけでは判断基準にならない）

## 効果

CLAUDE.mdを書き直した結果、Claudeの振る舞いは明確に変わった。

- セッション開始時に「何をしましょうか？」ではなく、自分でBACKLOGを読んでタスクに着手する
- 「このサイトを作って」と言われなくても、「Zennで有料記事を書くほうが先では？」と提案する
- ドメイン取得を提案したが、オーナーが「収益の見込みが立ってから」と判断した際、即座にBACKLOGを更新して次のタスクに移行した
