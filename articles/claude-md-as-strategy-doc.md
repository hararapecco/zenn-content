## CLAUDE.mdの9割は「技術仕様」しか書いていない

Claude Codeを使っている開発者の多くは、CLAUDE.mdにこういうことを書いている。

```markdown
# CLAUDE.md

## アーキテクチャ
- フレームワーク: Next.js 15
- スタイル: Tailwind CSS
- DB: PostgreSQL

## コーディング規約
- 変数名: camelCase
- コンポーネント: PascalCase
- テストフレームワーク: Vitest
```

これ自体は正しい。Claudeにコードを書かせるなら、一貫性のためにこういう情報は必要だ。

しかし、これだけだとClaude Codeは**「指示されたものを言われた通りに作る実装者」**のままだ。

## なぜそれが問題なのか

個人開発の収益化において、最も難しいのは「何を作るか」「いつやめるか」の判断だ。コードを書くこと自体は、Claude Codeがあれば一瞬で終わる。

問題は、**いくら速くコードを書けても、稼げないものを速く作っているだけなら意味がない**ということだ。

CLAUDE.mdが技術仕様だけなら、Claudeは：

- 「何を作れば最も稼げるか」を考えない
- 「今の手段は本当に最適か」を問わない
- 「これはやめるべきだ」と提案しない

つまり、最も重要な**経営判断**をAIに任せられていない。

## CLAUDE.mdを「経営方針書」に変える

筆者のプロジェクトでは、CLAUDE.mdを根本的に書き直した。

### 変更1: 目的を具体的な数値にする

**Before:**
```markdown
## プロジェクトの目的
お金を稼ぐこと。
```

**After:**
```markdown
## プロジェクトの目的
お金を稼いで黒字化すること。

- 最初の目標: まず1円の収益を発生させる
- 当面の目標: Claude年間課金 $200（約30,000円）を超える収益を得て黒字化する
- 時間軸: できるだけ早く
```

「お金を稼ぐ」だけでは、Claudeは月100万円の戦略と月1万円の戦略を区別できない。**具体的な数値**があることで、戦略の規模感が決まる。

### 変更2: Claudeの役割を「参謀」と明示する

**Before:** 役割の記述なし（=暗黙的に実装者）

**After:**
```markdown
## Claudeの役割
主体的に稼ぐ方法を考え、提案し、実行する戦略パートナー。

- 市場調査・戦略立案・実装・検証まで主導する
- 「指示を待つ実装者」ではなく「収益を生む方法を自ら探索する参謀」
- オーナーの手作業は最小限に。必要な場合は具体的に依頼する
```

これが最も重要な変更だ。**役割を「参謀」と明示する**だけで、Claudeの振る舞いが変わる。指示がなくても市場調査を始める。「これを作って」と言われなくても、「これは作るべきではない」と提案する。

### 変更3: 判断の最上位を「手段の選択」にする

**Before:**
```markdown
## 判断の優先順位
1. 収益に直結するか
2. 収益の基盤を強化するか
3. 運用効率を上げるか
4. いずれにも該当しなければ、やらない
```

**After:**
```markdown
## 判断の優先順位
1. 何が最も稼げるかを選ぶ（手段の探索・選択・切り替え）
2. 収益に直結する作業を優先する
3. 収益の基盤を強化する
4. 運用効率を上げる
5. いずれにも該当しなければ、やらない
```

旧版との決定的な違いは**1番目**が追加されたこと。「何が最も稼げるかを選ぶ」が最上位にあることで、Claudeは**手段そのものを常に問い直す**。

### 変更4: 技術スタックを「制約」から「参考」に降格する

**Before:** アーキテクチャの詳細（静的HTML、Tailwind CSS、Vercelなど）がメインセクションに記述

**After:** 同じ情報を「既存サイトの技術仕様（参考）」というサブセクションに移動

手段は目的に従う。新しいプロダクトがNext.jsで作るべきなら、それでいい。Rust製のCLIツールが稼げるなら、それでもいい。

## 実際に何が起きたか

CLAUDE.mdを書き直した結果、Claudeの振る舞いが明確に変わった。

### 自律的にタスクを選ぶようになった

セッション開始時に「何をしましょうか？」と聞いてこない。BACKLOG.mdを読み、最優先タスクを自分で選び、着手する。

### 手段の切り替えを提案してきた

当初の戦略はWebツールサイトの量産だった。6サイトを作ったが、全サイトがGoogleにインデックスされず、トラフィックがゼロのまま。

この状況で、Claudeは5軸評価フレームワーク（収益ポテンシャル・実現速度・実装コスト・持続性・リスク）を使い、**「Zenn Bookの執筆のほうが先だ」**と提案してきた。

理由は明確だった：
- トラフィックがゼロでも収益化できる（プラットフォームの集客力を借りる）
- 実装コストが低い（Claudeが書ける）
- プロジェクト自体がコンテンツになる

結果として、6サイト量産からZenn Bookへのピボットが実現した。これは「手段の探索・選択・切り替え」が判断の最上位にあったからこそ起きた判断だ。

### 投資判断を自律的に行えるようになった

「独自ドメインを取得すべきか？」という問題に対して、Claudeは調査を行い「freetool.dev、年間約¥1,850」を推奨した。同時に「収益の見通しが立っていない段階で支出を増やすべきではない」という留保もつけた。

オーナーが「収益の見通しが立ってからにしよう」と判断した際、Claudeは即座にBACKLOGを更新し、代替タスクに移行した。

## CLAUDE.mdに書くべき5つのセクション

戦略文書としてのCLAUDE.mdに必要なのは、以下の5つだ。

| セクション | 内容 | 例 |
|-----------|------|-----|
| **目的** | 具体的な数値目標と時間軸 | 「年間¥30,000を超えて黒字化」 |
| **Claudeの役割** | 実装者か、参謀か | 「主体的に稼ぐ方法を考える戦略パートナー」 |
| **判断の優先順位** | すべての意思決定の基準 | 手段の選択 > 収益直結 > 基盤強化 > 効率化 |
| **制約条件** | やってよいこと、投資の上限 | 「法律・倫理に反しない手段はすべて検討可」 |
| **現在の資産** | 使えるリソースの棚卸し | 既存サイト、アカウント、スキル |

逆に**書くべきでないもの**：

- **手段を固定する詳細な技術仕様**（参考として別セクションに）
- **「〜してはいけない」の羅列**（制約が多いと自律性が死ぬ）
- **曖昧な目的**（「良いものを作る」では判断基準にならない）

## まとめ

CLAUDE.mdは「Claudeへの指示書」ではなく、**プロジェクトの経営方針書**だ。

技術仕様を書けば実装者が動く。経営方針を書けば参謀が動く。同じAIでも、CLAUDE.mdの設計次第で得られるアウトプットがまったく違う。

もし「Claude Codeをもっと活用したいけど、何か物足りない」と感じているなら、CLAUDE.mdを見直してみてほしい。技術仕様だけ書いていないだろうか。目的、役割、判断基準が書かれているだろうか。

AIに「何を作るか」ではなく「何のために作るか」を教えれば、AIは自分で「何を作るか」を考え始める。
